# v3.5 Gateway 异步任务与飞书结果回传方案 PRD

## 一句话定义

将 `ios-shortcuts-gateway` 从“移动端同步等待结果”的模式升级为“移动端只负责提交任务、电脑后台执行、飞书异步回传结果”的模式，避免 iOS 快捷指令超时，并保留完整的源视频链接供后续处理。

## 背景

`本地 HTTP + Tailscale` 路线已经验证通过：

- iPhone 可访问 Gateway `/health`
- Gateway 可正常接收并执行 `clip` / `analyze`
- 但 `clip` / `analyze` 为长任务，iOS 快捷指令等待最终结果会频繁超时

因此，需要将用户交互改为：

- 提交即返回 `ACCEPTED`
- 电脑后台继续执行
- 执行完成后主动把结果发回飞书

## 目标

1. iPhone 快捷指令只负责提交任务，不等待最终剪藏/拆解完成。
2. Gateway 在接收任务后立即返回 `request_id`。
3. Gateway 后台执行 `Clipper / Analyzer`。
4. Gateway 在任务结束后主动向飞书发送结果消息。
5. 飞书回传中必须包含原始视频链接，便于用户后续处理。
6. 保留本地 debug 目录与结构化状态文件，便于排查。

## 非目标

1. 不让 iPhone 成为长任务轮询主界面。
2. 不把原始 debug 内容直接发到飞书。
3. 不在这一版引入复杂队列中间件。
4. 不做多用户权限系统。

## 新链路

`iPhone 快捷指令 -> Gateway -> 立即返回 ACCEPTED -> 后台执行 Clipper / Analyzer -> 飞书结果回传`

## 用户体验

### 提交阶段

快捷指令立即显示：

- 任务已提交
- `request_id`
- 结果将稍后通过飞书返回

### 完成阶段

飞书返回：

- 成功 / 部分完成 / 失败 / 登录态失效
- 原始视频链接
- 规范化链接
- 原始分享文本
- clipping note 路径
- breakdown note 路径
- `request_id`

## 关键要求

### 1. 异步提交

`POST /short-video/task` 默认返回：

- `status = ACCEPTED`
- `request_id`

而不是等待最终业务结果。

### 2. 后台状态持久化

每个请求目录至少保留：

- `request.json`
- `status.json`
- `clipper-result.json`
- `analyzer-result.json`
- stdout / stderr logs

### 3. 飞书结果回传

Gateway 在任务进入最终状态时主动发送飞书消息。

最终状态包括：

- `SUCCESS`
- `PARTIAL`
- `FAILED`
- `AUTH_REQUIRED`

### 4. 必须回传的源信息

飞书消息中必须包含：

- `source_url`
- `normalized_url`
- `original_source_text`

如果三者都存在，则三者都返回。

## 飞书回传消息最小字段

- `request_id`
- `action`
- `status`
- `message_zh`
- `source_url`
- `normalized_url`
- `original_source_text`
- `clipper_note`
- `analyzer_note`
- `failed_step`
- `auth_action_required`
- `refresh_command`

## 安全边界

飞书消息中不应包含：

- cookies 内容
- storage state 内容
- 原始堆栈
- 完整 debug 包内容

## 当前建议

正式版应将“移动端同步等待结果”彻底降级为调试模式，将“提交任务 -> 飞书异步回传”作为默认工作流。这样更符合长任务实际情况，也更接近真实使用。
